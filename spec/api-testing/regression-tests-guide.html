TalentLock Regression Tests

What the automated regression suite does, how it is organised, and how to run it.
Generated: July 2026  |  Branch: cursor/regression-tests-9a23

In short: Regression tests automatically re-check that TalentLock still works correctly after every code change. They cover API security, marketplace flows, billing limits, AI features, messaging, dashboards, frontend utilities, and legacy validation scripts — catching bugs before they reach production.

1. What is a regression test?

A regression test verifies that existing behaviour has not broken after a change. In TalentLock, the suite runs hundreds of automated checks against the API server, frontend components, and (optionally) a live browser.

Each test follows a simple pattern:

  1. Arrange — set up inputs, auth tokens, or database fixtures.
  2. Act — call an API endpoint, render a component, or load a page.
  3. Assert — confirm the response status, JSON shape, UI text, or security header matches expectations.

If a developer changes code that breaks a previously working feature, the matching test fails immediately — in local development or in GitHub Actions CI — before the change is merged.

3.5 Legacy validation bridge — 28 scripts

Location: scripts/test-regression.mjs  |  Command: pnpm run test:validate

Before the Vitest suite existed, TalentLock used standalone validate-*.mjs scripts run manually. The bridge orchestrator still runs all 28 legacy scripts for backwards compatibility. Each script is mapped to a modern Vitest suite in artifacts/api-server/tests/regression/legacy-map.ts.

4. What regression tests protect against

  • Security regressions — users accessing other users' bookings, agreements, or messages (IDOR)
  • Auth bypass — admin routes without CSRF, missing security headers
  • Billing bugs — quota limits not enforced; wrong 402 error shape
  • Data integrity — profile completeness score wrong; vault listing incomplete profiles
  • API contract drift — response shapes changing without notice
  • Frontend UX bugs — wrong error handling on plan limits; broken pagination
  • Feature regressions — agreements, AI, messaging, notifications breaking silently

6. Local setup (secrets never committed)

  1. Copy .env.example → .env locally.
  2. Fill in: DATABASE_URL, CLERK_SECRET_KEY, CLERK_PUBLISHABLE_KEY.
  3. Optional: ADMIN_PASSWORD, CSRF_SECRET, demo Clerk user IDs.
  4. Run pnpm run test:integration to execute all 147 integration tests.

.env and .env.test are gitignored. Never commit secrets to the repository.

10. How a typical integration test works

  1. Test file calls integrationEnvReady() — skips entire suite if secrets missing.
  2. apiClient.ts mints a real Clerk JWT for demo employer or freelancer.
  3. Supertest sends HTTP request to the in-process Express app.
  4. Response status, JSON body, and headers are asserted.
  5. For quota tests: asserts HTTP 402 with { error, code, planNeeded } shape.
  6. For IDOR tests: same route called as wrong user → expects 403.
